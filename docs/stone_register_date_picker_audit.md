# Stone Register Date Picker Migration Matrix

The HTML mockup is a skin and interaction reference only. It does not define
canonical month data, conversion rules, persistence formats, validation, route
behavior, or date-only semantics.

The shared picker was introduced first, followed by low-risk reusable wrappers
and bounded wrapper consumers. Complex calendar, event, recurrence, Day View,
Ma'at flow, and journal navigation surfaces remain deferred until their
preservation contracts are written and tested.

### Event Create Dialog Date Preservation Contract

- Entry point: `_openCalendarScopedNoteDialog` in `lib/features/calendar/calendar_page.dart`, specifically the local `datePicker()` used by the shared-calendar Add note/create-event dialog.
- Scope boundary: only the event-create date picker is migrated in this pass. Day sheet date picker, Day View navigation, Ma'at flow date picker, recurrence-until, reminder repeat end-date, Flow Studio, shared-calendar persistence semantics, and journal navigation remain out of scope.
- Current picker API: the dialog invokes `EventCreateDatePicker.show(context: dialogCtx, initialDate: KemeticMath.toGregorian(selYear, selMonth, selDay), initialMode: dateMode)` and mutates local state only after a non-null Done result.
- Date model and defaults: the dialog continues to store the selected event date as Kemetic `selYear`/`selMonth`/`selDay`; the picker returns a Gregorian date-only value plus the final visible Gregorian/Kemetic mode; initial selection still comes from `_today`.
- Display and conversion: Gregorian labels use app Gregorian month names; Kemetic labels use `getMonthById`; conversion still uses `KemeticMath.fromGregorian` and `KemeticMath.toGregorian`; Done updates the dialog label to the selected visible mode.
- Validation: Gregorian month/leap-day clamping, Kemetic epagomenal 5/6 clamping, and the old 401-year windows centered on the dialog initial date remain in the picker; title validation and start/end time adjustment remain in `saveNote`.
- Persistence path: shared-calendar event creation still calls `_saveSingleNoteOnly` with the same title/location/detail, calendar id/name, all-day flag, start/end time, selected color, category, alert offset, and selected `selYear`/`selMonth`/`selDay`; `_saveSingleNoteOnly` still owns local note insertion, Supabase upsert, alert filing, and shared-calendar item-added fanout.
- Cancel/Done/reopen: Cancel returns null and leaves the event draft unchanged; Done saves exactly the visible selected date into `selYear`/`selMonth`/`selDay`; reopening shows the saved selected date; no off-by-one date after save/reopen.
- Surrounding draft state: title, location, detail, all-day/timed state, start/end duration behavior, calendar ownership, alert, category, color, save button state, and dialog cancel behavior must survive picker open/cancel/done.
- Existing tests: `test/widgets/event_create_date_picker_test.dart` covers picker open/cancel/done/reopen, mode conversion, no off-by-one, surrounding draft labels, and small-phone layout; `test/features/calendar/event_create_date_picker_migration_test.dart` guards production routing, save payload inputs, Day sheet deferral, and the audit contract; `test/features/calendar/shared_calendar_item_added_notifications_test.dart` guards shared-calendar event fanout through `_saveSingleNoteOnly`.

### Flow Studio Start/End Preservation Contract

- Entry points: `_pickGregorianDate`, `_pickKemeticDate`, `_pickRangeStart`, `_pickRangeEnd`, `_pickComposeRangeStart`, and `_pickComposeRangeEnd` in `lib/features/calendar/calendar_flow_studio_page.dart`.
- Current public/local API: private picker helpers return `Future<DateTime?>`; build and compose callers own all state mutation after a non-null result.
- Date model and defaults: all returned values are Gregorian date-only `DateTime`; picker seeds from the current selected value or `DateTime.now()`; build end defaults to `_endDate ?? _startDate`; compose end defaults to `_composeEndDate ?? _composeStartDate`.
- Display formatting: build and compose labels continue through `_fmtGregorian` and `_fmtKemetic`; Kemetic month names continue through `getMonthById`.
- Validation: Gregorian month/leap-day clamping, Kemetic epagomenal 5/6 clamping, and existing `_rebuildSpans` start/end order normalization must remain unchanged.
- Persistence path: date picking only mutates local Flow Studio state; draft persistence, `_FlowStudioResult`, AI import initialization, planned-note payloads, and calendar scheduling remain caller-owned.
- Cancel/Done/reopen: Cancel returns null and does not mutate build or compose ranges; Done saves exactly the visible selected date through existing caller normalization; reopening shows the saved selected value.
- Dirty state and side effects: build date changes keep `_dateRangeEditedInCurrentEditor` and `_applySelectionToDrafts`; compose date changes keep `_composeManualDateRangeEdited`, duration fallback, and `dateOnlyForAiFlow`; AI generation still uses `resolveAiFlowDateRange`.
- Existing tests: `test/features/calendar/flow_studio_state_sync_regression_test.dart` already covers draft state, range labels, system toggles, generated save payloads, import initialization, route results, generated schedule spans, and mobile toolbar layout. This migration must add focused picker tests for open, Cancel, Done, reopen, range validation preservation, Compose behavior, no off-by-one dates, and small-phone overflow.

### Recurrence Until Preservation Contract

- Entry point: note editor `End Repeat` → `On Date...` inside `lib/features/calendar/calendar_page.dart`, calling `RecurrenceUntilDatePicker.show` with `initialDate: endDate ?? gDay.add(Duration(days: 30))`, `allowPast: false`, `firstDate: gDay`, and `lastDate: gDay.add(Duration(days: 365 * 10))`.
- Scope boundary: this contract covers the note recurrence-until picker only. Reminder repeat end-date selection has a separate contract and shares the wrapper without sharing note recurrence save logic.
- Date model and defaults: picker returns Gregorian date-only `DateTime?`; caller sets `endType = NoteRepeatEndType.onDate` and `endDate = picked` only for non-null results.
- Display and conversion: the picker opens in Gregorian mode, supports Gregorian/Kemetic mode switching through app `KemeticMath` and `getMonthById`, clamps Gregorian month lengths and Kemetic epagomenal days, and keeps the selected date stable across mode switches.
- Validation: the selected until date remains constrained to the first occurrence through ten years after it; recurrence count and end-type selection stay outside the date picker.
- Persistence path: note recurrence saves still flow through `_saveRepeatingNoteAsHiddenFlow`, `_buildNoteRuleDates`, `generateNoteRecurrenceDates`, `CalendarPageState.ruleToJson`, and hidden-flow/event materialization; event start/end duration, calendar ownership, alerts, colors, title/location/details, and shared-calendar behavior are not changed.
- Cancel/Done/reopen: Cancel returns null and leaves repeat settings unchanged; Done returns the visible selected date for valid selections; reopening uses the saved selected until date; no off-by-one date after save/reopen.
- Existing tests: `test/features/calendar/recurring_scope_behavior_test.dart` covers repeating-scope integration guards, `test/features/calendar/shared_calendar_item_added_notifications_test.dart` guards the hidden repeating-note save fanout, and this pass must add focused recurrence-until picker and recurrence generation coverage.

### Reminder Repeat End Date Preservation Contract

- Entry point: reminder editor repeat `End date` action inside `_openReminderEditor` in `lib/features/calendar/calendar_page.dart`.
- Scope boundary: only the repeat end-date picker is migrated in this pass. The reminder start date, event create date wheels, Day sheet date picker, Day View navigation, Ma'at flow date picker, shared-calendar event logic, journal navigation, Flow Studio, and note recurrence save logic remain out of scope.
- Current picker API: the reminder caller now invokes `RecurrenceUntilDatePicker.show(sheetCtx, initialDate: endLocal ?? startLocal, allowPast: true)`; the wrapper keeps its public `DateTime?` result API and the caller owns all mutation.
- Date model and defaults: `endLocal` is a Gregorian date-only `DateTime?`; `null` displays as Never and saves as no repeat end date; the picker seeds from the current `endLocal` or `startLocal`.
- Date-only/time-zone behavior: picker results are normalized with `DateUtils.dateOnly`; no timezone conversion, recurrence serialization, or scheduling normalization is changed.
- Display formatting: the surrounding reminder editor keeps `MM/DD/YYYY` labels and the End date action sheet; Gregorian/Kemetic picker labels continue through shared app calendar utilities.
- Validation: caller-owned clamping remains unchanged, so a picked end date before `startLocal` becomes `DateUtils.dateOnly(startLocal)`; repeat kind, interval, weekdays, Gregorian month targets, decan targets, and Kemetic month targets remain outside the picker.
- Persistence path: saving still builds `ReminderRule(endLocal: repeat.kind == ReminderRepeatKind.none ? null : endLocal)`, previews locally, bumps reminder data version, calls `_upsertReminderRule`, and leaves reminder persistence plus notification sync caller-owned.
- Cancel/Done/reopen: canceling the action sheet or picker leaves `endLocal` unchanged; Clear sets `endLocal` to null; Done saves exactly the visible selected date before the existing start-date clamp; reopening shows the saved end date; no off-by-one date after save/reopen.
- Small-phone layout: the shared Stone Register sheet must not overflow when opened from the reminder editor footer area.
- Existing tests: `test/widgets/recurrence_until_date_picker_test.dart` covers reminder-style open/cancel/done/reopen, existing past end-date display, caller-owned start clamp, surrounding labels, and small-phone layout; `test/features/calendar/recurring_scope_behavior_test.dart` guards the `_openReminderEditor` routing/save path; `test/features/reminders/reminder_rule_test.dart` guards reminder repeat end-date payload serialization.
